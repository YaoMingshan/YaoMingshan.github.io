性能

开发了MoE kernel，并且用了fp8数据类型，相比于bf16，MoE提升了3.5倍，其中1.5倍是精度降低带来的，3.5/1.5是开发kernel带来的收益。

Microscaling data formats

是由OCP定义的数据类型，link.

为了减少大模型训练的计算时间，会采用算力更高的低精度数据类型进行训练，但是低精度数据类型的动态范围小，如果直接将高精度度直接转成低精度，可能表达不了，因此要做数据缩放，缩放到低精度能动态范围内，才可以做训练。

将它们转换为较窄位宽的格式（例如 8 位或更少）会引入不可接受的舍入误差，除非对值进行适当的缩放。例如，大型模型中的某些权重可能是 0.0001、0.0005 或 -0.0007，但简单地将它们转换为 FP8 会将它们全部舍入为同一个数字：零，因为 FP8E4M3 中可表示的最小正值约为 0.0019。

为了解决这个问题，通常的做法是为每个张量应用一个缩放因子，该因子会重新缩放张量，同时将其值保持在目标数据格式的可表示范围内。这确保了可用的动态范围得到充分利用。例如，如果一个张量的值都在 -0.0001 到 0.0001 之间，则其缩放因子可以设置为 4,480,000。这将导致缩放后的张量值处于 [-448, 448] 范围内，这对应于 FP8E4M3 的可表示范围。

微缩放更进一步，它对张量的细粒度block应用缩放，而不是对整个张量使用单一缩放因子。微缩放 (MX) 格式通过定义一组低精度、微缩放的数据格式来标准化此方法。

如果我们使用具有 E4M3 元素类型的 MXFP8 格式，则数据将由 FP8E4M3 元素组成，并将 FP8E8M0 比例因子应用于连续的 32 元素块，如下图所示。

Blackwell上的挑战

cuda core成为瓶颈

介绍一下fp8反量化是什么样子，以deepseek v3为例，A 矩阵以大小 1×128 的块进行缩放，B 矩阵以大小 128×128 的块进行反量化。流程是：

  • dot计算reduceK 128的中间结果tmp
  • 在temp上做缩放，乘以a_scale, b_scale
  • 再将temp的值累加到res上

在hopper上面，tensor core的输出在寄存器上，cuda core可以直接在寄存器上做scaling，不需要数据搬运；但是blackwell上，tensor core的输出在tmem上，需要将数据搬运到寄存器上然后cuda core做scaling。

scaling算法上，只需要dot计算量的1/32（以32缩放），但是blackwell上tensor core fp8是4,500 TFLOP/s，但是fp32 cuda core是80 TFLOP/s，两者比值为56。此时cuda core编程bound了，tensor core只能发挥57%的算力。

fp8量化耗时太长

在用于 MoE 训练的分组矩阵乘法中：

  • M=131,072
  • K=7,168
  • N=2,048

131,072×7168×2048×2 FLOP=3.85 TFLOP

Blackwell GPU（B200s）的 FP8 矩阵乘法吞吐量约为 3,300 TFLOP/s。这意味着执行矩阵乘法的预期时钟时间为：

3.85 TFLOP / 3,300 TFLOP/s=1.16 ms

  • Load A: 131,072×7,168×2B=1.88 GB
  • Load B: 7,168×2,048×2B=0.029 GB
  • Store quantized A: 131,072×7,168×1B=0.94 GB
  • Store quantized B: 7,168×2,048×1B=0.015 GB
  • Store scale factors for A: 131,072×7,168×132×1B=0.029 GB
  • Store scale factors for B: 7,168×2,048×132×1B=0.0005 GB

2.9 GB / 6.5 TB/s≈0.44 ms

也就是说，fp8量化和MoE计算是大概1:2的关系，但是有可能带宽用不到那么满，所以fp8量化时间会更长。

低精量化算法

选择了fp8e3m4作为量化数据类型，按照32 block做量化，算法如下：

找到V中绝对值最大的值，将其除以448，转化成fp8e8m0，作为scale；将v除以s，转化成fp8，作为量化值。

  • BF16 (or FP32) vector V={Vi∣i=0,…,31}
  • Corresponding FP8E4M3 vector Q={Qi∣i=0,…,31}
  • FP8E8M0 scale S

S= cast_to_fp8e8m0(absolute_max (V) / 448)

Qi= cast_to_fp8e4m3 (Vi / S)

Blackwell新feature - block scale MMA

blackwell tensor core中支持dot + scale，但是必须要用tcgen05.mma…block_scale指令，其中scale需要放到tmem中。

首先，tcgen05.mma 指令只需要一个线程即可异步启动。这与 Hopper 不同，Hopper 中的 wgmma 指令需要整个 warpgroup（128 个线程）才能异步启动。因此，我们必须摒弃 Hopper 内核中常见的生产者-消费者模式。

其次，tcgen05.mma 支持 2-CTA 矩阵乘法，其中两个 SM 通过共享 B 矩阵协作执行矩阵乘法。这减少了内存流量和共享内存使用量，从而支持更深的矩阵乘法流水线。我们的基准测试表明，与非集群版本相比，此 2-CTA 模式可为 MXFP8 矩阵乘法带来约 15% 至 20% 的加速。

第三，如前所述，tcgen05.mma 将结果累加到 TMEM 中，而不是寄存器中。虽然这减少了寄存器压力，但它也通过 tcgen05.ld 和 tcgen05.st 指令在 TMEM 和寄存器之间引入了额外的数据移动。我们必须确保最大限度地减少这些数据移动。

最后，tcgen05.mma 指令要求block scale驻留在 TMEM 中。然而，没有直接的方法将比例因子从 HBM 加载到 TMEM。最快的方法是先使用 cp.async.bulk.tensor 指令（利用张量内存加速器，即 TMA）将数据从 HBM 加载到片上 SMEM，然后使用 tcgen05.cp 指令将其从 SMEM 传输到 TMEM。为了实现此方法，比例因子必须存储在 tcgen05.mma 期望的布局。

用1个block 3个wrapgroup(12 warp)，异构的形式做pipeline，包含：

  • 两个warp group将tensor core的结果从tmem -> register -> smem –tma-> global
  • 一个warp group做输入数据搬运以及mma
    • warp0搬运 input
    • warp1搬运 scale
    • warp2 tcgen05.cp smem -> tmem
    • warp3（cta0）发起mma，两个协同cta只需要有个一发起tcgen05.mma.cta_group::2.mxf8f6f4.block_scale
  • 各个流水线都用mbarrier做同步

性能如何呢？

  • 非block scale dot，tmem用满128 * 512，性能达到3,200 T；如果只用128*256，下降到 3,040 T
  • block scale dot，由于pipeline流水线开销，达到2,750 T，是bp16的1.8倍

L2 cache optimization中讲了一个有意思的点，ThunderKittens 内核的一种启发式算法，它通过确保所有 148 个 SM 在任何给定时间计算的输出矩阵区域尽可能呈正方形，从而最大限度地提高 L2 的复用率。
